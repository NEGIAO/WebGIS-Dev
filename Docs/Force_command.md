# 🛠️ WebGIS 开发维护日志规范 (Agent 强制执行)

本目录用于记录 WebGIS 项目从 3.0 全栈版本起的所有开发日志、Bug 修复及功能改进。本规范旨在确保在 AI 辅助开发过程中，代码的可回溯性、架构的清晰度以及团队协作的高效性。

---

## ⚠️ Agent 执行准则

1.  **强制性记录**：任何对 `frontend/` 或 `backend/` 文件夹下的代码进行**增加、删除、修改**的操作，必须在本目录下生成对应的维护日志。
2.  **及时性要求**：日志必须在**每次对话任务结束前**创建并提交，严禁跨日合并或事后补录。
3.  **权限收束 (决策权)**：Agent **严禁自作主张**进行 `git commit`、`git stash` 或 `git push` 操作。所有版本控制的最终决策权归属于用户。Agent 仅负责准备变更内容。
4.  **文档收束 (禁止乱建 README)**：Agent 禁止在 `/docs` 以外的任何位置（如根目录或子目录）新建临时的 README 或说明文档。所有关于更新内容的说明、操作指南必须统一维护在 `/docs` 的日志文件中。
5.  **结构同步 (核心指令)**：每次任务涉及文件增删或目录调整后，**必须强制同步更新**以下三个 `README.md` 中的文件结构树（Tree），并附带简洁的功能注释：
    * 项目根目录 `./README.md`
    * 前端目录 `./frontend/README.md`
    * 后端目录 `./backend/README.md`

---

## 📝 日志模板

**文件名格式**：`YYYY-MM-DD-更新主题.md`  
*(例如：`2026-04-19-fix-auth-loading.md`)*
存放路径：`/Docs/yy-mm-dd/yyyy-mm-dd-更新主题.md`

**日志必须包含以下信息：**

* **日期和时间**：精确到分钟（例如：2026-04-19 14:30）。
* **修改内容**：简明扼要地列出本次任务完成的具体功能开发或 Bug 修复。
* **修改原因**：说明本次变动的背景、解决的痛点或优化的动机。
* **影响范围**：受此修改影响的系统模块（如：鉴权系统、底图链路、数据库、URL 参数管理等）。
* **修改的文件路径**：列出所有发生变动的文件绝对路径，便于回溯。

---

## 📜 归档管理与审计

* **强制执行者**：由 GitHub Copilot、Gemini 或其他 AI 协作助手强制执行。
* **合规性检查**：任何跳过日志记录、拒绝同步文件结构、擅自创建冗余说明文件或越权执行 Git 命令的操作都将被视为**违规操作**。
* **版本演进**：本规范自 WebGIS 3.0 正式启用，所有重大演进必须在此留下“数字足迹”，相同的日期，大版本只能更新一次如 3.0.1、3.0.2，同一天的多次更新仍归于大版本的更新日志内。
