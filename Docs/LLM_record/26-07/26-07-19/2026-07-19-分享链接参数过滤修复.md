# 2026-07-19 分享链接参数过滤修复

## 基本信息
- **日期时间**: 2026-07-19 14:00
- **修改类型**: Bug 修复 / 逻辑重构
- **严重程度**: 高（分享链接曾泄露用户私有参数 ut/loc/p）

## 事件逻辑链分析

### 核心症状
点击 TopBar「分享」按钮生成的分享链接中，`ut`（用户身份）、`loc`（定位授权来源）、`p`（GPS 编码位置）三个用户私有参数未被清除，导致分享链接携带单用户敏感信息。同时原 `buildShareMarkedUrl` 和 `syncShareFlagInCurrentUrl` 还存在 `cs`（罗盘）仅在 compass 禁用时才删除的条件 Bug。

### 根因
旧逻辑采用**白名单**方案（仅保留 6 个参数），过于激进地清除了 `cv`（Cesium 相机姿态）、`cs`（罗盘位置，启用时）等恢复 3D 视角必需的关键参数，导致 Cesium 分享链接无法还原完整的相机视角和罗盘状态。

同时在上一轮重构中错误地移除了 `compassStore` 引用，使 `cs` 的条件保留逻辑完全失效。

### 受影响模块
- `frontend/src/components/Shell/TopBar.vue` — `buildShareMarkedUrl()`、`syncShareFlagInCurrentUrl()` 及关联辅助函数

### 解决方案
改用**黑名单**方案：明确只清空 `ut`、`loc`、`p` 三个用户私有参数，其余全部原样保留。

| 参数 | 处理方式 | 原因 |
|------|---------|------|
| `ut` | ❌ 清空 | 用户身份标识（guest/admin/registered），单用户独有 |
| `loc` | ❌ 清空 | 定位授权来源（gps/ip），单用户独有 |
| `p` | ❌ 清空 | GPS 精准位置编码，单用户独有 |
| 其余所有参数 | ✅ 保留 | 不影响隐私，且有助于还原完整视图状态 |

## 修改内容
1. **重写 `buildShareMarkedUrl()`**：白名单 → 黑名单模式，遍历 `SHARE_EXCLUDED_PARAMS = ['ut', 'loc', 'p']` 逐个删除，其余所有参数原样保留
2. **重写 `syncShareFlagInCurrentUrl()`**：同样采用黑名单模式，与 `buildShareMarkedUrl` 保持逻辑一致，分享后同步清理当前地址栏
3. **删除死代码**：移除 `normalizePositionCode()`、`resolvePositionCodeForShare()`、`extractPositionCodeFromText()` 三个不再需要的辅助函数（共 ~35 行）；移除未使用的 `normalizeLocationFlag`、`useCompassStore` import 及 `compassStore` 引用
4. **降级路径修复**：catch 分支不再拼接 `loc=0&p=...`，改为仅追加最小标记 `s=1&l=<default>`

## 修改文件路径
1. `frontend/src/components/Shell/TopBar.vue` — 脚本部分（第 312-668 行）重构

## 影响范围
- **分享链接生成**：点击「分享」后生成的 URL 不再包含 ut/loc/p 参数
- **当前页 URL 同步**：分享成功后 `syncShareFlagInCurrentUrl()` 同步清理当前地址栏
- **Cesium 分享**：`cv` 参数恢复保留，3D 分享链接可完整还原相机视角
- **罗盘分享**：`cs` 参数始终保留（URL 有什么就保留什么）
- **降级兼容**：`layer` 别名 → `l` 归一化逻辑保留，历史链接兼容
- **向后兼容**：接收方打开分享链接的启动流程（router guard → urlParamStore）无需改动

## 测试方案
1. 在 OL 2D 视图下点击「分享」，验证生成的链接无 ut/loc/p，有 lng/lat/z/l/s/view 及其他原有参数
2. 在 Cesium 3D 视图下点击「分享」，验证生成的链接包含 cv 参数，可还原相机姿态
3. 罗盘未启用时点击「分享」，验证 cs 不出现在链接中（URL 本身就没有 cs）
4. 罗盘启用后点击「分享」，验证 cs 正常出现在链接中
5. 分享成功后检查当前地址栏 URL，确认 ut/loc/p 已被清理
6. 用分享链接在新窗口打开，验证视图状态正确还原

## 性能指标
无性能影响。黑名单遍历 3 个键（ut/loc/p）的开销可忽略不计，与原白名单遍历 5 个键 + 重建 URLSearchParams 相比，新方案直接在原 hashParams 上删除，减少了对象创建。
