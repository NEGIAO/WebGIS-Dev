# v3.2.2 变更日志

- **日期和时间**：2026-06-04 01:30
- **修改内容**：
  1. 新增 `httpStatusMap.js` 统一 HTTP 状态码映射模块（标准 HTTP 码 + 高德 infocode）
  2. 重构 axios 响应拦截器错误处理，所有 HTTP 错误码附带中文描述
  3. 统一 6 个 API 模块的状态码处理（geocoding / weather / ipLocation / map / locationSearch / client）
  4. 修复日志监控面板（LogMonitor）将 5xx/4xx HTTP 错误误判为正常 INFO 日志的 Bug
  5. 瓦片请求生命周期重写：fetch() 替代 img.src，AbortController 真正中断 TCP 连接
  6. 修复 useBasemapResilience 的 validateBaseLayerSwitch setTimeout 泄漏

- **修改原因**：
  1. 前端无统一 HTTP 状态码映射，错误处理分散在多个文件，各自硬编码
  2. 后端高德 API Key 未配置时返回 503，前端无法从错误信息中识别具体原因
  3. LogMonitor 的 `getLogClass()` 仅检查 `ERROR`/`INFO` 等关键字，uvicorn 访问日志中 `INFO` 在行首优先命中，行尾的 `503` 被忽略，显示为绿色正常日志
  4. `tileLifecycle.ts` 通过 `img.src` 发起瓦片请求，但 `AbortController.signal` 未绑定到网络请求，`abort()` 无法中断底层 TCP 连接，被墙的源占据 6 个浏览器并发连接槽位 30-60 秒

- **影响范围**：前端 API 层（`src/api/`）、日志监控面板（`LogMonitor`）、瓦片源工厂（`tileSource/`）、底图容灾（`useBasemapResilience`）

- **优化解决方案**：

  ### 1. 统一状态码映射

  - 新增 `httpStatusMap.js`：`HTTP_STATUS_MAP`（80+ 个标准 HTTP 码）、`AMAP_INFOCODE_MAP`（40+ 个高德 infocode）、`getHttpStatusMessage()`、`getAmapErrorMessage()`、`buildHttpErrorMessage()`、`classifyHttpStatus()`、`isRetryable()` 等辅助函数
  - `client.js` 拦截器：错误日志格式升级为 `[Backend API] [503 服务暂不可用] 后端未配置高德服务密钥`，`apiError` 新增 `status`/`statusText` 字段
  - `handleApiError()`：UI 通知自动附加 `[503 服务暂不可用]` 标签
  - 6 个 API 模块的 catch 块统一使用 `getAmapErrorMessage()` 替代内联硬编码

  ### 2. 日志监控 HTTP 状态码识别

  - `LogMonitor.vue` 的 `getLogClass()` 函数新增正则 `/\b([1-5]\d{2})\b/g` 检测
  - 匹配优先级：`ERROR/FAILED` → HTTP 5xx（红）→ HTTP 4xx（黄）→ HTTP 2xx（绿）→ `WARN` → `SUCCESS` → `INFO`
  - 取行内最后一个匹配（状态码在行尾），避免端口号干扰

  ### 3. 瓦片请求 AbortController 修复

  - `tileLifecycle.ts` 新增 `fetchTileAsBlobUrl()` 函数：用 `fetch()` + `AbortController.signal` 加载瓦片
  - `prioritizeTileSourceRequest()` 重写：fetch → blob URL → img.src，abort 时立即释放 TCP 连接
  - `abortTileSourceRequests()` 四层级联：epoch++ → `controller.abort()` → 标记 tile → `source.clear()`
  - blob URL 在 abort/error 时通过 `URL.revokeObjectURL()` 及时释放
  - `TILE_REQUEST_TIMEOUT_MS`（15s）超时保护，防止被墙请求无限挂起

  ### 4. Timer 泄漏修复

  - `useBasemapResilience.js` 的 `validateBaseLayerSwitch()` 新增 `timeoutId` 变量
  - `settle()` 和 `cleanup()` 中均调用 `clearTimeout(timeoutId)`

- **测试方案**：
  1. 日志监控：触发 503 错误 → 日志行显示为红色（非绿色）
  2. 日志监控：正常请求 → 200 日志行显示为绿色
  3. 瓦片切换：加载被墙底图 → 切换 → 旧请求立即 `(canceled)` → 新请求不排队
  4. 瓦片超时：限速 1Kb/s → 15 秒后触发降级
  5. 正常底图：切换无回归，Console 无报错
  6. 快速连续切换 5-10 次 → 最终底图正确显示，无未捕获异常
