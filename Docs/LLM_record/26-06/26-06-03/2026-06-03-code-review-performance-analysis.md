## 2. 影响响应速度的关键参数分析

### 2.1 参数总览

| 参数 | 文件 | 当前值 | 影响程度 | 说明 |
|------|------|--------|----------|------|
| **MAX_CONCURRENCY** | rectify.py:24 | 8 | 🔴 高 | 并发请求数 |
| **REQUEST_DELAY** | rectify.py:160 | 0.1s | 🔴 高 | 请求间隔 |
| **MAX_RETRIES** | fetch.py:12 | 2 | 🟡 中 | 重试次数 |
| **timeout.connect** | fetch.py:21 | 10s | 🟡 中 | 连接超时 |
| **timeout.read** | fetch.py:21 | 15s | 🟡 中 | 读取超时 |
| **z 阈值** | rectify.py:266 | 9 | 🟡 中 | 纠偏起始级别 |
| **GCJ2WGS_MAX_ITERATIONS** | transform.py:38 | 20 | 🟢 低 | 迭代次数 |

### 2.2 关键参数详解

#### 🔴 MAX_CONCURRENCY = 8（高影响）

**作用**：限制同时进行的 HTTP 请求数

**影响分析**：
```
z=17 级别：1 个目标瓦片 → 2×2 = 4 个源瓦片
前端请求 20 个瓦片 → 20 × 4 = 80 个源瓦片
并发 8 → 需要 80/8 = 10 批次
```

**响应时间估算**：
- 单个请求耗时：~200ms（网络）+ 100ms（处理）= 300ms
- 10 批次 × 300ms = **3 秒**（不含延迟）

**优化建议**：
```python
# 如果服务器性能好，可以增加到 12-16
MAX_CONCURRENCY = 12

# 如果仍然超时，降低到 4-6
MAX_CONCURRENCY = 4
```

#### 🔴 REQUEST_DELAY = 0.1s（高影响）

**作用**：每个请求之间的延迟，避免限流

**影响分析**：
```
80 个源瓦片 × 0.1s = 8 秒额外延迟
总响应时间 = 3s（网络）+ 8s（延迟）= 11 秒
```

**问题**：这个延迟是在任务创建时设置的，但实际上：
1. 缓存命中的瓦片不需要延迟
2. 并发已经限制了请求数，不需要额外延迟

**优化建议**：
```python
# 方案1：完全移除延迟（依赖并发控制）
REQUEST_DELAY = 0

# 方案2：只对缓存未命中的请求添加延迟
# 在 _get_tile_cached 内部处理

# 方案3：减小延迟
REQUEST_DELAY = 0.02  # 20ms
```

#### 🟡 MAX_RETRIES = 2（中影响）

**作用**：请求失败时的重试次数

**影响分析**：
```
超时重试：每次重试等待 0.5s, 1.0s
最坏情况：2 次重试 × 1.5s = 3s 额外延迟
```

**优化建议**：
```python
# 如果网络稳定，可以减少到 1 次
MAX_RETRIES = 1

# 如果网络不稳定，保持 2 次
MAX_RETRIES = 2
```

#### 🟡 timeout.connect = 10s（中影响）

**作用**：建立 TCP 连接的超时时间

**影响分析**：
```
正常情况：连接建立 ~100ms
超时情况：等待 10s 后失败
```

**优化建议**：
```python
# 如果服务器响应快，可以减少到 5s
timeout=httpx.Timeout(connect=5.0, read=10.0, write=10.0, pool=5.0)
```

#### 🟡 z 阈值 = 9（中影响）

**作用**：低于此级别直接返回源瓦片，不执行纠偏

**影响分析**：
```
z <= 9：1 次请求，~200ms
z > 9：4-9 次请求，~1-3s
```

**优化建议**：
```python
# 可以提高到 10 或 11，减少纠偏请求
if z <= 10:
    return source_tile
```

#### 🟢 GCJ2WGS_MAX_ITERATIONS = 20（低影响）

**作用**：坐标转换的最大迭代次数

**影响分析**：
```
通常 5-10 次已收敛，CPU 耗时 <1ms
20 次迭代：~2ms（可忽略）
```

**优化建议**：
```python
# 可以减少到 10 次，精度已足够
GCJ2WGS_MAX_ITERATIONS = 10
```

## 3. 响应时间模型

### 3.1 单瓦片请求（z=17，无缓存）

```
请求流程：
1. 坐标转换：~1ms
2. 计算源瓦片范围：~1ms
3. 获取 4 个源瓦片：
   - 并发 4 个请求：~200ms（网络）
   - 请求间隔：4 × 0.1s = 400ms（如果启用）
4. 合并瓦片：~50ms
5. 裁剪：~20ms
6. 返回：~10ms

总计：~680ms（有延迟）/ ~280ms（无延迟）
```

### 3.2 批量请求（20 瓦片，z=17，无缓存）

```
请求流程：
1. 20 个瓦片并行处理
2. 每个瓦片需要 4 个源瓦片
3. 总共 80 个源瓦片请求

并发 8，无延迟：
- 批次：80/8 = 10 批
- 耗时：10 × 200ms = 2s

并发 8，有延迟（0.1s）：
- 额外延迟：80 × 0.1s = 8s
- 耗时：2s + 8s = 10s
```

### 3.3 缓存命中场景

```
缓存命中：~5ms（文件读取）
缓存未命中：~200-1000ms（网络请求）
```

# fetch.py

## 6. 总结

### 关键发现

1. **REQUEST_DELAY 是最大的性能瓶颈**：0.1s × 80 个请求 = 8 秒额外延迟
2. **MAX_CONCURRENCY 影响批次数量**：从 8 增加到 16 可以减少一半批次
3. **缓存命中率决定实际性能**：缓存命中时响应 <10ms

### 响应时间对比

| 场景 | 当前配置 | 优化后 |
|------|----------|--------|
| 单瓦片（无缓存） | ~680ms | ~280ms |
| 批量 20 瓦片（无缓存） | ~10s | ~3s |
| 单瓦片（有缓存） | ~5ms | ~5ms |
