## 问题事件逻辑链条分析（先分析后改动）
- 核心症状：后端任务处理进度可见，但 GeoTIFF 文件从后端传输到浏览器阶段缺乏实时 UI 进度，蓝色传输条静止。
- 根本原因：
  1. 组件使用 fetch 手动读取流，未统一复用 download API 层能力。
  2. 现有 Axios 客户端响应拦截器默认返回 data，下载响应头（content-disposition/content-length）无法稳定透传到组件。
  3. 任务状态芯片仅绑定后端任务状态，未表达“传输中/传输失败”前端阶段状态。
- 受影响模块：
  1. 前端下载 API 封装（src/api/download.js）
  2. 前端 Axios 基座（src/api/backend.js）
  3. 下载面板组件（src/components/MapDownloader.vue）
- 可选优化方案：
  1. 在 apiDownloadTaskFile 中引入 onDownloadProgress 回调并统一百分比计算。
  2. 对 blob 响应跳过 data 解包，保留完整 AxiosResponse 供组件读取响应头与文件名。
  3. 组件侧改为调用 API 层下载，并在回调中同步 transferProgress/bytes。
  4. 传输异常时标记“传输失败”，暴露重试入口。
- 最终实施方案：
  1. 重构 apiDownloadTaskFile(taskId, onProgress, options?)。
  2. MapDownloader 接入 apiDownloadTaskFile + onProgress，下载完成后自动触发浏览器保存。
  3. 增加 transferStatus 计算优先级：transferring > transfer-failed > backend status。
  4. 保持 5 分钟 timeout，不改后端 download.py。

- [frontend/src/components/MapDownloader.vue](frontend/src/components/MapDownloader.vue)：改为 Axios 下载链路，实时传输进度、文件大小、失败重试和文件名提取。

## 影响范围
- 前端下载导出链路与 UI 反馈。
- 不涉及后端下载任务计算逻辑。

## 性能与稳定性预期
- 大文件下载阶段具备可观测进度，不再“无 UI 黑盒等待”。
- 失败可重试，减少误判为系统卡死的体验问题。


## 修改内容（实施完成）
- 将下载文件接口重构为 apiDownloadTaskFile(taskId, onProgress, options)，并使用 onDownloadProgress 回调实时输出百分比与字节信息。
- MapDownloader 组件改为调用 API 层下载，不再直接 fetch 读取流。
- 文件命名优先读取 content-disposition，缺省为 map-export-
${taskId}.tif。
- 传输状态新增“传输中/传输失败”覆盖显示逻辑，失败时保留重试按钮。
- 轮询阶段停止后端 success 自动二次下载，避免 store 与组件并发触发文件拉取。

## 优化解决方案
- 前端生成阶段与传输阶段分离：后端状态条只展示任务进度，蓝色条展示真实网络传输进度。
- Axios 二进制响应保留完整响应对象，确保可读取响应头并驱动文件命名。
- 错误捕获统一落到 transferState.error，UI 可直接看到失败原因并重试。

## 测试方案
- 执行 npm run build，构建通过。
- 验证后端任务 success 后自动进入传输阶段，进度条实时更新。
- 验证传输失败时状态显示“传输失败”并可再次点击“重新下载到本地”。
