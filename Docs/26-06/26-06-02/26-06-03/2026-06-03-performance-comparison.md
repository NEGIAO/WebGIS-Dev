## 2. 详细对比

### 2.1 缓存读取路径

**优化前**：
```
文件 → read() → Image.open() → image.load() → image_to_bytes() → 返回
         ↓           ↓              ↓              ↓
       磁盘I/O    创建对象      像素解码       PNG编码
```

**优化后**：
```
文件 → read_bytes() → 返回
         ↓
       磁盘I/O
```

**性能分析**：
- 优化前：1次文件读取 + 1次 PIL 解码 + 1次 PNG 编码
- 优化后：1次文件读取
- **预期提升**：缓存命中场景减少 50%+ 的 CPU 和 I/O 开销

### 2.2 瓦片格式验证

**优化前**：
```python
image = bytes_to_image(tile_bytes)  # PIL 完整解码
png_bytes = image_to_bytes(image)   # PNG 编码
```

**优化后**：
```python
if _is_valid_image_bytes(tile_bytes):  # 魔数检查（O(1)）
    _save_tile_bytes(tile_path, tile_bytes)
```

**性能分析**：
- 优化前：完整图像解码+编码（毫秒级）
- 优化后：4-12 字节魔数检查（微秒级）
- **预期提升**：格式验证阶段减少 90%+ 耗时

### 2.3 错误恢复机制

**优化前**：
```python
async def fetch_tile(url, client=None):
    try:
        ...
    except Exception:
        return await fetch_tile(url, client=None)  # 递归！
```

**问题**：
- 每次重试增加调用栈深度
- 理论上可能导致栈溢出（极端情况）
- 事件循环错误时递归行为不可预测

**优化后**：
```python
for attempt in range(MAX_RETRIES + 1):
    try:
        ...
    except Exception:
        if is_event_loop_error and attempt < MAX_RETRIES:
            reset_async_client()
            continue
        raise
```

**优势**：
- 固定栈深度，无溢出风险
- 重试逻辑清晰可控
- 日志记录便于调试

## 4. 内存优化

### 4.1 单瓦片处理

**优化前**：
```python
# bytes_to_image() 创建 Image 对象
image = Image.open(BytesIO(tile_bytes))  # +2MB（像素数据）
image.load()                              # 强制加载到内存

# image_to_bytes() 创建 BytesIO 缓冲区
img_buffer = BytesIO()                    # +256KB
image.save(img_buffer, format="PNG")      # 编码
```

**优化后**：
```python
# 直接返回原始字节
return tile_path.read_bytes()  # 无额外内存分配
```

**内存节省**：约 2.3MB / 瓦片
