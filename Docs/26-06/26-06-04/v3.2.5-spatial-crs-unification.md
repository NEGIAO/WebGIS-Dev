# v3.2.5 空间分析统一 EPSG:3857 变更日志

- **日期和时间**：2026-06-04 23:40
- **修改内容**：
  1. 后端空间分析统一在 EPSG:3857 平面坐标系下执行，所有距离/大小参数统一为米
  2. `router.py` 统一收口 CRS 转换：输入 GeoJSON(4326) → 转 3857 → operation → 结果转回 4326
  3. 删除 `meters_to_degrees_approx` 近似转换函数，buffer/multiRingBuffer 直接在 3857 下用米
  4. aggregation 的 `grid_size` 从度数改为米（默认 500m）
  5. simplify 的 `tolerance` 从度数改为米
  6. 合并 `grid_size_meters`（fishnet 专用）到通用 `grid_size` 字段
  7. 修复前端 aggregation `_gridSize` 命名 bug（用户输入的网格大小被静默忽略，始终用默认值）
  8. 删除前端 `metersToDegrees`/`getReferenceLat` 重复函数

- **修改原因**：
  1. 原架构各操作坐标系不一致：buffer 用近似转换，aggregation 用度数，fishnet 用 pyproj
  2. `meters_to_degrees_approx` 在高纬度和大距离下误差显著
  3. 用户输入网格大小以米为单位更直观
  4. 前端 `_gridSize` 命名导致 UI 输入值被丢弃

- **影响范围**：后端空间分析全部操作、前端空间分析面板和调度逻辑

- **优化解决方案**：

  ### CRS 转换架构

  ```
  前端 → GeoJSON(4326) → router → pyproj 转 3857 → operation(纯 3857) → pyproj 转 4326 → 前端
  ```

  - `utils.py` 新增 `reproject_geoms_to_3857()` / `reproject_geoms_to_4326()` / `reproject_result_to_4326()`
  - `pyproj.Transformer` 使用 `@lru_cache` 缓存实例
  - operation 文件不直接依赖 pyproj，只在 3857 坐标下工作

  ### 各操作改动

  | 操作 | 改动前 | 改动后 |
  |------|--------|--------|
  | buffer | radius_m → meters_to_degrees_approx → buffer(deg) | radius_m 直接 buffer(m) |
  | multiRingBuffer | distances → meters_to_degrees_approx | distances 直接使用 |
  | aggregation | grid_size=0.01 度 | grid_size=500 米 |
  | simplify | tolerance=度 | tolerance=米 |
  | fishnet | 自带 pyproj 转换 | 删除 pyproj，由 router 统一处理 |
  | voronoi | 退化 fallback 0.01 度 | 退化 fallback 1000 米 |

  ### 前端改动

  - `SpatialAnalysisPanel.vue`：aggregation 标签"度"→"米"，gridSize 默认 0.01→500，min/max/step 调整
  - `useSpatialAnalysis.js`：删除 `metersToDegrees`/`getReferenceLat`/`toLonLat` import；simplify 直接传米；aggregation 用 `gridSize` 替代 `_gridSize`

- **测试方案**：
  1. buffer(1000m) → 结果缓冲区在地图上约 1km
  2. aggregation(grid_size=500) → 方格网每格约 500m
  3. simplify(tolerance=100) → 节点减少，几何形状保持
  4. fishnet(grid_size=500) → 渔网每格约 500m
  5. 高纬度区域测试 → 无明显变形（pyproj 精确转换）
