# 2026-05-05 底图切换阻塞请求中断修复

- 日期和时间：2026-05-05 17:21

## 事件逻辑链条分析
- 核心症状：用户切到受限/高延迟 XYZ（如 Google）后，再切回可用底图（高德/天地图）时，新底图请求明显排队，地图长时间空白，通常需刷新页面恢复。
- 根本原因：
  1. 旧图源请求虽然触发了 AbortController，但在 AbortError 分支没有将瓦片状态从 LOADING 释放。
  2. OpenLayers 全局瓦片队列存在并发槽位上限，LOADING 未释放会持续占槽，导致后续新图源请求被饿死。
  3. 底图观察器在释放旧图源时仅 clear/setSource(null)，未先显式中断网络请求，导致旧请求继续占用浏览器连接资源。
- 受影响模块：
  1. 前端瓦片请求工厂（source 级请求中断与错误状态处理）
  2. 底图切换 watcher（旧图源释放前的中断顺序）
- 优化方向：
  1. 将中断视为“必须释放队列项”的完成态处理。
  2. 在图源层引入请求代际（epoch）和请求超时，阻止旧请求回写。
  3. 在 watcher 释放旧 source 前先执行 abort，确保网络与队列双释放。

## 修改内容
- 在 `useTileSourceFactory.ts` 中新增：
  1. `TILE_REQUEST_TIMEOUT_MS` 超时控制（15s）。
  2. `markTileAsError`，在中断/失败/过期响应场景显式调用 `tile.setState(3)` 释放 OL 队列占位。
  3. `abortEpoch` 代际机制，防止旧请求在切换后回写。
  4. `abortTileSourceRequests` 中断时同步递增 epoch，并执行 `source.clear()` 清理残留缓存。
- 在 `useBasemapSelectionWatcher.js` 中新增：
  1. 引入 `abortTileSourceRequests`。
  2. 在 `releaseLayerSource` 内执行“先 abort，再 clear，再 setSource(null)”顺序。

## 修改原因
- 保证“当前用户选择”的底图始终拥有最高请求优先级，切换后旧请求立即失效，不再占用连接和队列槽位。

## 影响范围
- 模块：前端底图切换链路（图源工厂 + basemap watcher）。
- 业务：XYZ/WMS/WMTS 底图切换体验、弱网与受限网络下恢复速度。

## 优化解决方案
1. 统一中断语义：Abort 不再静默悬挂，而是显式释放队列状态。
2. 请求生命周期隔离：每次中断提升 `abortEpoch`，旧异步回调即使晚到也会被丢弃并释放。
3. 切换顺序修复：底图 watcher 在源释放前先 abort，避免“先断引用后仍在请求”的泄漏。

## 性能指标
- 本次为功能性稳定修复，未进行基准压测；预期收益：

  2. 弱网/受限网络下，恢复到可用底图的首屏时间稳定。

## 同日增量更新（17:41）- 隐藏图层源泄露根治

### 深度调试记录
- 现象：即使启用了 Abort 与全量中断策略，Google→Tianditu 切换仍需约 20 秒等待。
- 调试链路追踪：`switchLayerById()` → `refreshLayerInstances()` → `ensureLayerSourceById()`
- **关键发现**：`switchLayerById()` 将旧底图设置为 `layer.visible = false`，但**未清除 source**。`refreshLayerInstances()` 内部仅处理 `visible=true` 的图层，隐藏图层的 source 永远不被触及。因此隐藏的底图 source 继续后台请求，**持续占用浏览器连接槽位，导致可见底图请求饥荒**。

### 根治方案
- 在 `useMapState.js` 中新增 `clearLayerSourceForced(id)`：
  1. 获取图层及其 source。
  2. 执行 `source.clear()`（清理瓦片缓存）。
  3. 执行 `layer.setSource(null)`（断开引用）。
  4. 这样彻底断开 OL 内部与该 source 的关联，后台请求被迫停止。
- 在 `switchLayerById()` 中的 `refreshLayerInstances()` **前**，新增预清理循环：
  1. 遍历 `layerList` 中所有 `visible=false` 的底图图层。
  2. 对每个即将隐藏的底图执行 `clearLayerSourceForced()`。
  3. 确保所有隐藏底图都已断源再进行 refresh，彻底消除连接占用。

### 修复机制
```javascript
// 在 refreshLayerInstances 前执行：
layerList.forEach((item) => {
    if (!item.visible) {
        const cfg = configs.find((c) => c.id === item.id);
        const category = cfg?.category;
        const isBasemap = category === 'label' || ... || category === 'custom';
        if (isBasemap) {
            clearLayerSourceForced(item.id);  // 强制断源
        }
    }
});
refreshLayerInstances({ layerList, instanceMap, configs });
```

### 影响范围
- 解决"底图快速切换后仍长时间白屏"这一最直观用户痛点。
- 特别是在 Google→Tianditu/高德 这类"高延迟源→低延迟源"切换场景下，效果显著。
- 弱网/VPN/国外网络环境下恢复速度大幅提升。

## 同日深度优化（17:50）- OL 瓦片队列与 Fetch Abort 联动强化

### 问题诊断（深层根因分析）
- 现象：即使已实施前3层修复，Google→Tianditu 仍需 >10s 延迟
- **隐藏根因**：OpenLayers 内部的瓦片加载队列（tile queue）有并发槽位限制（通常 6-8 个）。当 `abortTileSourceRequests()` 被调用时：
  1. Fetch abort signal 被发送，但网络连接可能已建立
  2. Fetch promise chain 仍在进行中，最终调用 `markTileAsError()`
  3. 期间内，OL 认为这些 tiles 仍在 LOADING 状态，占用队列槽位
  4. 新底图的 tile 请求被饿死，无法获得槽位
- **关键发现**：OpenLayers 依靠 `tile.setState(3)` 调用来释放队列占位，而不是依赖 fetch abort

### 根治方案（三位一体）
#### A. 强化 OL 内部缓存遍历（markAllSourceTilesAsError）
- 新增函数，支持多种方式访问 OL source 的内部缓存：
  1. `tileCache_` / `tileCache`：OL 的主瓦片缓存（Map-like 结构）
  2. `tileLoadingKeys_`：当前加载中的 tiles 键值表
  3. `tiles_`：某些版本的 OL 在这里保存 tiles 数组
- 对每个找到的 tile 调用 `markTileAsError(tile)`，确保 OL 队列立即释放

#### B. 改进 Fetch Abort Signal 处理（tileLoadFunction）
- **关键改进**：在 signal 被 abort 时，立即在 listener 中调用 `markTileAsError(tile)`
  ```javascript
  const onAbortSignal = () => {
      try { markTileAsError(tile); } catch { }
  };
  signal.addEventListener('abort', onAbortSignal, { once: true });
  ```
- 这样不需要等待 fetch promise chain 完成，abort 发生的第一时刻就释放队列槽位
- Promise chain 的 catch 中仍保留 `markTileAsError()` 作为双保险

#### C. 强化 abortTileSourceRequests 的三级联动
```
Step 1: abortEpoch++  → 标记源代际，后续响应被丢弃
Step 2: controller.abort() → 发送 signal，listeners 立即触发 markTileAsError
Step 3: markAllSourceTilesAsError(source) → 遍历缓存，强制标记所有残留 tiles
Step 4: source.clear() → 清空缓存和队列
Step 5: 新建 AbortController → 准备下次切换
```

### 性能指标预期
- **绝对延迟降低**：从 >10s 降至 <1s（取决于网络延迟）
- **相对改善**：新底图请求获得连接槽位的时间从"旧请求完全超时后"提前至"abort signal 触发时"
- **最坏场景**：即使网络连接已建立，abort 仍会在瓦片级别释放 OL 队列，而不是在 HTTP 连接级别等待（这是浏览器层面无法完全控制的）
